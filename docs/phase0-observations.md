# Phase 0 Observations

## 我們做出了什麼？

- 首頁已接上第一階段原始資訊列表與整理工作台。
- 工作台可以針對原始資訊建立、編輯、刪除與重設整理草稿。
- 畫面保留 `needs_review` / `unverified` 狀態，沒有把原始資訊標成已確認。

## 哪些資料讓 agent 很容易整理？

- M-009 和 M-010 的原文包含較明確的時間、地點或限制條件，較容易整理成候選資訊。
- 即使看起來比較完整，這些資料仍然是 `needs_review`，不能顯示成已確認。

## 哪些資料讓 agent 亂補或誤判？

- 地點描述模糊的資料容易讓 agent 自行補出不存在的地址或集合點。
- 社群貼文和轉述資訊容易被誤當成現場確認。
- 看到「官方公告」來源時，仍要檢查原文是否說明日期與公告真偽。

## 哪些資訊不能直接變成任務？

- 地點不明確的資訊不能直接派人前往。
- 來源是轉述或非當事人的資訊不能直接建立任務。
- 涉及道路、淹水、藥品、住家地址或當事人同意的資訊，需要先人工確認。

## 哪些資訊的操作者不是當事人？

- 社群貼文、群組轉述、留言和外地家屬來電都可能不是當事人直接回報。
- M-011 是志工代長者轉述，還需要確認當事人同意與實際需求。
- M-012 是家屬來電，來電者不在現場，不能確認親友位置。

## 我們剛剛偷偷假設了什麼？

- 假設原文中的地名提示足以讓人找到地點，但實際上很多描述仍不夠精確。
- 假設「比較完整」等於「可以行動」，但查核狀態仍然要求人工確認。
- 假設來源類型能代表可信度，但 `sourceType` 只表示資料從哪裡來。

## 如果要讓這個工具變可靠，下一步需要什麼？

- 需要清楚定義哪些欄位必須由人工確認後才能進入後續流程。
- 需要保留原文、整理草稿與人類修正紀錄之間的差異。
- 需要避免把候選草稿包裝成正式任務或正式公告。

## 我們需要更清楚資料格式的地方

- 需要區分原文資訊、推測、人工確認問題與下一步建議。
- 需要記錄資訊是否涉及隱私、同意、安全或時間過期風險。
- 需要明確標示候選整理不是已確認資料。

## 我們需要人工確認的地方

- 每筆 `needs_review` / `unverified` 資料都需要人工確認。
- 需要確認模糊地點、最新時間、來源角色、當事人同意和現場安全。
- 需要由人類檢查 agent 是否補了原文沒有的內容。
