# 使用者訪談筆記

## 受訪 persona 回饋摘要

### 回報者

1. 我想完成的事
   - 盡快把我知道的現場情況送出去，讓其他人知道可能需要協助。
   - 希望能先留下不確定的資訊，而不是因為表單太難就放棄。
   - 也希望自己不必為了填寫完整而拖延，能先把已知的部分送出去。

2. 我看不懂的地方
   - 不知道哪些欄位是必填、哪些可以先留空。
   - 不清楚畫面上哪些內容只是「待確認」，哪些內容看起來像已經被確認。
   - 也不確定自己是否應該填上「我不知道」還是直接跳過某些欄位。

3. 我最怕的錯誤
   - 只是在轉述別人的說法，卻被系統誤認為親眼確認。
   - 不確定的地點、數量或需求被自動補成像真的。
   - 如果自己的不確定內容被視為事實，可能會讓後續協作者做出錯誤判斷。

4. 我不敢直接相信的資訊
   - 看起來很完整，但來源不清、地點不明、或是由他人轉述的資訊。
   - 即使內容聽起來合理，也不代表它已經被現場確認。

5. 我希望系統提醒我的事
   - 明確提示哪些內容是我不確定的地方。
   - 明確標示這段資訊不是我親眼看到的。
   - 如果我只能提供有限資訊，也希望系統能鼓勵我先提交，再由後續流程補齊。

6. 我想反問開發者的一個問題
   - 如果我只知道一半，系統會不會把我沒確認的部分誤當成事實？
   - 如果我只是轉述，系統是否有辦法讓我明確表達「這是別人說的」而不是「我親眼看到的」？

### 資訊整理者

1. 我想完成的事
   - 把混亂的原始資訊整理成下一位協作者能看懂的狀態，但又不想把不確定內容誤當成已確認。
   - 希望自己能快速判斷哪些資訊值得往下流，哪些資訊應該先停下來等待確認。

2. 我看不懂的地方
   - 不容易分辨哪些內容是回報者直接提供的原始資訊，哪些內容是整理後的判斷。
   - 有時候同一件事會有多個版本，這讓我很難判斷哪一個才應該被視為主要資訊。

3. 我最怕的錯誤
   - 模糊資料被直接變成任務。
   - 把回報者提供但尚未確認的內容，誤當成已確認事實。
   - 如果不同來源的說法彼此衝突，卻被整理成單一結論，後續協作者會更容易被誤導。

4. 我不敢直接相信的資訊
   - 來源不明、地點模糊、社群貼文或沒有實際查核的資訊。
   - 只要缺少時間、地點或當事人身份的上下文，我就會把它視為不夠可靠。

5. 我希望系統提醒我的事
   - 清楚標示來源是什麼、哪些內容仍需人工確認、哪些內容只是回報者提供的原始敘述。
   - 若有多筆說法或衝突內容，也希望能被明確提示，而不是被簡化成單一結果。

6. 我想反問開發者的一個問題
   - 如果一筆資訊看起來很完整，是否仍然要保留「待確認」狀態？
   - 如果不同來源提供不同版本，系統是否應該讓我看到這些差異，而不是隱藏掉？

### 行動者

1. 我想完成的事
   - 知道哪些資訊現在可以採取行動，哪些仍需要再確認，避免做出錯誤判斷。
   - 希望能快速區分「可以立即行動」、「需要再確認」、「不建議行動」這三種狀態。

2. 我看不懂的地方
   - 不確定哪些資訊已經被確認，哪些內容還是需要再確認。
   - 我也不確定某些資訊是「已確認可用」，還是只是「已被整理過但仍需人工複查」。

3. 我最怕的錯誤
   - 把未確認資訊當成已確認，然後跑到錯的地方或處理一筆已失效的需求。
   - 如果資訊過時或有衝突，我希望能在採取行動前就被提醒，而不是等到現場才發現。

4. 我不敢直接相信的資訊
   - 地點不清、時間不明、需求不確定，或來源不是當事人直接提供的資訊。
   - 如果缺少最新時間或現場狀態，我會把它當成可能已經變更的資訊。

5. 我希望系統提醒我的事
   - 提醒資訊是否可能過期、是否有衝突，以及是否需要再確認再行動。
   - 如果某筆資訊看起來很接近可行動，但仍有未確認欄位，我希望能被明確警示。

6. 我想反問開發者的一個問題
   - 如果一筆資訊還沒被確認，系統是不是應該更明確提醒「不要直接行動」？
   - 如果資訊可能已失效，系統是否應該主動提示我先停下再確認？

## 共同觀察

- 三種 persona 都非常在意「未確認」與「已確認」之間的差異。
- 他們都對「看起來完整但其實不確定」的資訊感到警戒。
- 現有原型已經保留了部分查核與待確認語意，但仍需要更清楚的視覺與文案提醒。
